# Sales Rep Copilot — RAG Project Roadmap
*Last updated: May 2026*

---

## Project Goal

A Streamlit web app where a sales rep types a natural language question about a B2B customer and gets a grounded, data-driven answer generated by Claude API — powered by a RAG pipeline over real transactional data.

**Example questions the system answers:**
- "Summarize customer ABC Corp"
- "Which Enterprise customers haven't ordered in 60 days?"
- "What is the RFM persona and contract status of customer X?"

---

Portfolio project to close 3 critical skill gaps simultaneously:
- RAG / embeddings
- MLflow experiment tracking

## Phase 1 — Data Preparation
**Status: Complete (2026-06-04)**

**Goal:** Load all CSVs, merge on cus_code, aggregate everything per customer into one clean row.

**Why:** RAG cannot work on raw database rows. One customer may have 500 transaction rows. You need one coherent summary per customer as the foundation for everything downstream.

**Actual Output:** Two tables in notebooks/data_prep.ipynb:
- `loads_enriched` — row-level loads joined with product and customer dimensions
- `loads_monthly` — loads_enriched aggregated to one row per cus_code + product + month

**Final embedding output (built in Phase 2):** loads_monthly itself — one row per customer × product family × month.

**Rules locked from this phase:**
- Loads filtered to 2023+ only — do not change this filter without explicit decision
- Enrich before aggregating — dimension columns must be available in the groupby
- loads_monthly is the embedding dataset — embed it directly at monthly granularity; do not roll up to customer level
- One src/ file per phase — data_prep.py owns Phase 1 logic only
- Never load excluded columns: cus_fin_code, cus_ext_ref, Segment_Abbreviation__c, RFMPersonas

## Phase 2 — Text Profile Generation
**Timeline:** Weekend 1, Day 1 afternoon

**Status: Complete (2026-06-04)**

**Goal:** Convert each loads_monthly row into a human-readable text string.

**Why:** This text is what gets embedded and stored. The quality of this text directly determines the quality of answers. Garbage in, garbage out.

**Example output:**
```
"Customer: PIERRE FABRE ΕΛΛΑΣ ΑΕ | Group: Pierre Fabre | Segment: Field Sales - South Greece |
Employees: 131 | Job Type: Chemicals & Plastic | Salesperson: Ioanna Kampouridou |
Product Family: EB | Month: 2023-01 | Revenue: €6,893 | Orders: 80"
```

**Output:** 89,799 text strings in data/processed/customer_profiles.csv — one per customer × product family × month.

**Rules locked from this phase:**
- data_prep.ipynb is responsible for saving loads_monthly.csv to data/processed/ — profile_builder must not re-run the raw pipeline
- profile_builder.py reads from data/processed/loads_monthly.csv as its sole input
- Text profile format is pipe-separated — do not change to natural language sentences without explicit decision
- customer_profiles.csv is the input to Phase 3 (embeddings) — column `text_profile` is what gets embedded

## Phase 3 — Embedding + Vector Store
**Timeline:** Weekend 1, Day 2

**Status: Complete (2026-06-05)**

**Goal:** Convert each text profile into a vector and store in ChromaDB.

**Why:** This is the Retrieval in RAG. When a user asks a question, the system finds semantically similar profiles — not keyword matches. This is the core technical skill every GenAI JD is asking for.

**Tools:**
- `sentence-transformers` — creates embeddings locally, free, no API cost
- `ChromaDB` — local vector database, zero infrastructure needed

**Output:** A persistent ChromaDB collection with all customer profiles embedded and indexed.

**Rules locked from this phase:**
- Embedding model is `all-MiniLM-L6-v2` — do not change without explicit decision (Phase 4 query embedding must use the same model)
- ChromaDB collection name is `customer_profiles`, path is `chroma_db/`, distance metric is cosine
- Vector IDs are `{cus_code}_{ProductFamily}_{month}` — deterministic, safe to rebuild
- Metadata fields per vector: cus_code, cus_name, segment, product_family, month, salesperson
- embeddings.py is idempotent — skips insert if collection is populated; `--force-rebuild` rebuilds from scratch
- embeddings.py reads from `data/processed/customer_profiles.csv` — do not change input source

## Phase 4 — RAG Pipeline
**Timeline:** Weekend 2, Day 1

**Goal:** Build the question → retrieve → answer loop.

**Step by step:**
1. User types a question in Streamlit
2. Question gets embedded using the same sentence-transformers model
3. ChromaDB finds top-N most semantically similar customer profiles
4. Retrieved profiles passed as context to Claude API
5. Claude generates a grounded answer
6. Answer displayed in Streamlit UI

**Why:** This is the Augmented Generation in RAG. Claude doesn't hallucinate because it receives real retrieved data as context. This is the core architectural pattern every GenAI role requires.

**Output:** A working Streamlit app where you can ask questions and get answers.

## Phase 5 — MLflow + Docker
**Timeline:** Weekend 2, Day 2

**Goal:** Add experiment tracking and containerize the app.

**MLflow — what to track:**
- Embedding model used
- Number of chunks retrieved (top-N)
- Question and answer pairs
- Retrieval latency

**Docker — what to containerize:**
- The full Streamlit app
- ChromaDB collection
- All dependencies in requirements.txt

**Why it matters:** Separates a notebook experiment from a production system. Closes two CV gaps in one afternoon. Answers the interview question "how would you deploy this?"

**Output:** A Dockerized app + MLflow experiment log — both screenshottable for portfolio.
